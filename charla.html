React en 5 minutos

DOM · Virtual DOM · Imperativo → Declarativo

🔧 Vanilla JS — Imperativo

Le dices al browser cómo hacerlo, paso a paso

HTML 
<button id="btn">Clic</button>
<span id="count">0</span>
JavaScript 
let count = 0;
const btn = document.querySelector('#btn');
const span = document.querySelector('#count');

btn.addEventListener('click', () => {
  count++;
  span.textContent = count; // vos movés el DOM
});
index.html
0
clics — actualizás el DOM vos mismo
El problema: estás mezclando lógica de estado con manipulación del DOM. A medida que la app crece, se vuelve difícil de leer y mantener.

⚛️ React — Declarativo

Le dices qué mostrar — React decide cómo

HTML 
<!-- una sola línea — React toma el control -->
<div id="root"></div>
JSX 
function Counter() {
  const [count, setCount] = useState(0);
  return (
    <button onClick={() => setCount(count + 1)}>
      Clics: {count}
    </button>
  );
}
ReactDOM.createRoot(document.getElementById('root'))
        .render(<Counter />);
Counter.jsx
re-renders: 0
React mantiene una copia ligera en memoria (Virtual DOM). Cuando el estado cambia, compara el árbol nuevo con el anterior (diff) y solo toca el DOM real donde algo cambió. La analogía: un arquitecto que hace planos antes de construir, no un obrero que rompe paredes a ciegas.

El problema

¿Por qué React solo no alcanza?

React es una librería UI

❌ Lo que React NO resuelve

  • Routing entre páginas
  • Renderizado en el servidor
  • SEO (el HTML llega vacío a Google)
  • Fetching de datos estructurado
  • Code splitting automático
  • Optimización de imágenes
  • Configuración del bundler

✅ Lo que Next.js agrega

  • File-based routing integrado
  • SSR, SSG, ISR, CSR por ruta
  • HTML pre-renderizado para crawlers
  • Fetch con cache y revalidación
  • Code splitting por ruta automático
  • Componente <Image> optimizado
  • Zero-config con Turbopack
Cuando Google rastrea una SPA de React, recibe un div vacío. JavaScript todavía no corrió, el bot no puede leer el contenido, tu e-commerce no aparece en búsquedas. Next.js resuelve eso sin configuración.

¿Qué es Next.js?

Contexto e historia

Next.js en contexto

Definición

Framework full-stack construido sobre React, mantenido por Vercel. Opinionated pero flexible.


Librería vs Framework

Librería (React) — te da herramientas, vos decidís cómo usarlas.

Framework (Next.js) — tiene opiniones: define la estructura, vos rellenás los detalles.

  • 2016 — Vercel lanza Next.js 1.0
  • 2019 — SSG nativo (Next 9)
  • 2020 — ISR, mejoras de performance
  • 2021 — Middleware, Next 12
  • 2022 — App Router beta (Next 13)
  • 2023 — App Router estable (Next 13.4)
  • 2024 — Next 14 · Server Actions estables
  • 2026 — Next 16
La distinción librería vs framework importa: Next.js ya tomó 50 decisiones por vos (estructura de carpetas, bundler, routing). Vos solo te concentrás en tu producto.

Estrategias de Rendering

El corazón de Next.js

4 formas de renderizar

Estrategia ¿Cuándo se genera el HTML? Ideal para SEO
SSG — Static Build time (una vez) Blogs, docs, portafolio ✅ Perfecto
ISR — Incremental Build + revalida cada N segundos E-commerce, noticias ✅ Muy bueno
SSR — Server En cada request Datos en tiempo real, auth ✅ Bueno
CSR — Client En el browser del usuario Dashboards privados, SPAs ⚠️ Limitado
Cada ruta de tu app puede tener una estrategia diferente. La portada SSG, el perfil SSR, el dashboard CSR — todo en el mismo proyecto.
La pregunta clave: ¿cuándo se genera el HTML? SSG: al construir. ISR: al construir, con fecha de vencimiento. SSR: en cada petición. CSR: el browser lo construye — eso es lo que hacía React solo.

SSG — Static Site Generation build time


// app/blog/[slug]/page.jsx

export async function generateStaticParams() {
  const posts = await getPosts()
  return posts.map(p => ({ slug: p.slug }))
}

// Componente async — el await ocurre en el servidor, no en el browser
export default async function BlogPost({ params }) {
  const post = await getPost(params.slug)
  return <article>{post.content}</article>
}

El HTML existe antes de que alguien lo pida. Sin carga en el servidor en runtime — solo archivos estáticos.

No hay useEffect, no hay useState, no hay loading manual. El componente es async y simplemente espera los datos. Esto es posible porque corre en el servidor en build time, no en el browser.

Metadata API — SEO sin configurar build time


// app/blog/[slug]/page.jsx — mismo archivo que el componente

export async function generateMetadata({ params }) {
  const post = await getPost(params.slug)
  return {
    title: post.title,
    description: post.excerpt,
    openGraph: {
      title: post.title,
      images: [post.cover],
    },
  }
}

export default async function BlogPost({ params }) {
  const post = await getPost(params.slug)
  return <article>{post.content}</article>
}
Google recibe el HTML con <title> y <meta> correctos — sin JavaScript, sin react-helmet, sin imports extra. Una función. Para SSG corre en build time; para SSR en cada request.
La charla vendió SEO como beneficio central — acá está cómo se implementa. generateMetadata va en el mismo archivo que el componente. Next.js genera el head completo. Si solo recordás una cosa del SEO en Next.js, es esta función.

SSR — Server Side Rendering por request


// app/perfil/page.jsx

import { cookies } from 'next/headers'

export default async function PerfilPage() {
  const token = cookies().get('token')?.value

  const res = await fetch('https://api.app.com/me', {
    headers: { Authorization: `Bearer ${token}` },
    cache: 'no-store', // sin cache = HTML fresco en cada petición
  })
  const user = await res.json()

  return <h1>Hola, {user.name}</h1>
}

El servidor genera el HTML para cada visita. Ideal cuando el contenido depende del usuario autenticado.

No podemos pre-generar el HTML en build time porque no sabemos quién va a entrar. Cookies, sesiones, datos en tiempo real — este es el caso.

ISR — Incremental Static Regeneration revalidación


// app/productos/page.jsx

export default async function ProductosPage() {
  const res = await fetch('https://api.tienda.com/products', {
    next: { revalidate: 60 }, // regenerar el HTML cada 60 segundos
  })
  const products = await res.json()

  return (
    <ul>
      {products.map(p => <li key={p.id}>{p.name}</li>)}
    </ul>
  )
}
SSG + fecha de vencimiento. El HTML se sirve estático — Next.js lo regenera en background cada N segundos. Sin rebuilds completos.
Un e-commerce con 50.000 productos no puede usar SSR puro ni SSG puro. ISR resuelve eso: velocidad estática con datos frescos. Un solo parámetro en el fetch — eso es todo.

App Router

File-based routing y layouts

La estructura de archivos ES el router


app/
├── layout.jsx        # Layout raíz
├── page.jsx          # Ruta: /
├── about/
│   └── page.jsx      # Ruta: /about
├── blog/
│   ├── page.jsx      # Ruta: /blog
│   └── [slug]/
│       └── page.jsx  # Ruta: /blog/:slug
└── (dashboard)/
    ├── layout.jsx    # Layout anidado
    ├── overview/
    │   └── page.jsx  # Ruta: /overview
    └── settings/
        └── page.jsx  # Ruta: /settings

Archivos especiales

  • page.jsx — la UI de la ruta
  • layout.jsx — wrapper persistente
  • loading.jsx — skeleton automático
  • error.jsx — error boundary
  • not-found.jsx — 404 por segmento
  • route.js — API endpoint
[slug] → parámetro dinámico
(nombre) → agrupa rutas sin afectar la URL
loading.jsx → Next.js lo muestra automáticamente mientras page.jsx resuelve el await — sin escribir lógica de loading
No hay que instalar ni configurar un router. Creas el archivo en la carpeta correcta y la ruta existe. El loading.jsx es especialmente útil: Next.js muestra ese skeleton automáticamente mientras cargan los datos — sin escribir una línea de lógica de loading.

Layouts anidados


// app/layout.jsx — presente en todas las rutas
export default function RootLayout({ children }) {
  return (
    <html lang="es">
      <body>
        <Navbar />
        {children}
        <Footer />
      </body>
    </html>
  )
}
Los layouts no se re-renderizan al navegar entre rutas del mismo segmento. Se montan una vez — la sidebar no parpadea, el scroll se preserva.
En Pages Router cada navegación re-renderizaba el layout completo. En App Router solo cambia la parte que realmente cambió. Experiencia mucho más fluida.

Server vs Client Components

El modelo mental más importante de Next.js moderno

Dos mundos, una app

🍳
Server Component

La cocina del restaurante — secretos, DB, APIs. El cliente nunca accede.

→
🍽️
Client Component

La mesa del cliente — interactivo. El usuario toca, hace click, escribe.

Server Component por defecto

  • Corre SOLO en el servidor
  • Accede a DB, secrets, filesystem
  • No llega al bundle JS del cliente
  • No puede usar hooks ni eventos

Client Component "use client"

  • Corre en el browser
  • Puede usar useState, useEffect
  • Maneja eventos (onClick, onChange)
  • Accede a APIs del browser
Regla práctica: empieza siempre como Server Component. Agrega "use client" solo cuando necesites interactividad real. La mayoría de los componentes de una app son estáticos — no necesitan JavaScript en el cliente.

Server Component en acción


// app/usuarios/page.jsx — Server Component (sin "use client")

// Este fetch corre en el servidor. API_KEY nunca llega al browser.
export default async function UsuariosPage() {
  const res = await fetch('https://api.ejemplo.com/users', {
    headers: { 'x-api-key': process.env.API_KEY },
    next: { revalidate: 30 },
  })
  const users = await res.json()

  return (
    <section>
      <h1>Usuarios</h1>
      {users.map(u => <div key={u.id}>{u.name}</div>)}
    </section>
  )
}

Sin useState, sin useEffect, sin loading manual. El componente es async — el HTML llega completo al browser.

Variables sin prefijo (API_KEY) → solo servidor, nunca llegan al browser.
Variables con NEXT_PUBLIC_ → se exponen al cliente. Usar solo para datos que pueden ser públicos.
Este es el patrón central de Next.js moderno. Antes: useEffect + useState + loading + error = 20 líneas. Ahora: un await. Y la API key nunca se expone al cliente. Variables sin NEXT_PUBLIC_ son privadas de servidor — clave para no filtrar secrets.

Client Component en acción


// components/BuscadorUsuarios.jsx
'use client'

import { useState } from 'react'

export function BuscadorUsuarios({ usuarios }) {
  const [query, setQuery] = useState('')

  const filtrados = usuarios.filter(u =>
    u.name.toLowerCase().includes(query.toLowerCase())
  )

  return (
    <div>
      <input
        value={query}
        onChange={e => setQuery(e.target.value)}
        placeholder="Buscar usuario..."
      />
      {filtrados.map(u => <div key={u.id}>{u.name}</div>)}
    </div>
  )
}

Los datos llegan como props desde el Server Component padre. La interactividad vive solo donde se necesita.

El patrón: el Server Component trae los datos y los pasa como props. El Client Component maneja la interactividad. Los datos fluyen del servidor al cliente — no al revés.

Server Actions & Route Handlers

El backend integrado

Server Actions — Mutations sin API


// app/contacto/page.jsx
import { redirect } from 'next/navigation'
import { revalidatePath } from 'next/cache'

async function enviarMensaje(formData) {
  'use server' // corre SOLO en el servidor

  await db.mensajes.create({
    data: {
      nombre: formData.get('nombre'),
      mensaje: formData.get('mensaje'),
    },
  })
  revalidatePath('/mensajes') // limpia la caché — la lista se actualiza
  redirect('/gracias')
}

export default function ContactoPage() {
  return (
    <form action={enviarMensaje}>
      <input name="nombre" required />
      <textarea name="mensaje" required />
      <button type="submit">Enviar</button>
    </form>
  )
}

El formulario habla directamente con el servidor — sin endpoint manual. Funciona incluso sin JavaScript en el cliente.

Next.js genera el endpoint automáticamente por debajo. La lógica y la DB nunca se exponen al cliente. Esto es lo que hace que Next.js sea realmente "full-stack".

Route Handlers — API REST nativa


// app/api/productos/route.js

import { NextResponse } from 'next/server'

export async function GET() {
  const productos = await db.productos.findMany()
  return NextResponse.json(productos)
}

export async function POST(req) {
  const body = await req.json()
  const nuevo = await db.productos.create({ data: body })
  return NextResponse.json(nuevo, { status: 201 })
}
¿Cuándo usarlos? Cuando una app móvil u otro cliente externo necesita consumir tu API. Para apps web, las Server Actions son más simples.
Server Actions: mutations desde tu propia app. Route Handlers: exponer una API pública para otros clientes — mobile, terceros, webhooks.

Optimizaciones integradas

Performance out-of-the-box

Lo que viene incluido

📸 <Image> — Imágenes inteligentes

  • Lazy loading automático
  • Conversión a WebP/AVIF según el browser
  • Genera srcset responsive
  • Previene Cumulative Layout Shift (CLS)

🔤 <Font> — Fuentes sin layout shift

  • Self-hosting automático de Google Fonts
  • Cero peticiones externas en runtime
  • font-display: swap por defecto

🔗 <Link> — Prefetching inteligente

  • Prefetch de rutas visibles en el viewport
  • Navegación instantánea para el usuario

⚡ Turbopack + Code Splitting

  • Bundler en Rust — hot reload ~10× más rápido
  • Cada ruta descarga solo su JavaScript
  • Lazy loading con dynamic()
Estos son problemas que los devs seniors pasaban horas configurando hace 5 años. Ahora vienen de fábrica. El componente Image especialmente — una imagen mal optimizada puede doblar el tiempo de carga de una página.

¿Cuándo usar Next.js?

Y cuándo quizás no

Casos de uso

✅ Ideal

  • E-commerce
  • Blogs y CMS
  • SaaS apps
  • Portales con auth
  • Landing pages con SEO
  • Tu portafolio personal

⚠️ Evaluar

  • Apps solo para usuarios auth (sin SEO)
  • Proyectos muy simples — Vite puede bastar
  • Prototipo rápido de una idea

❌ Probablemente no

  • Apps 100% offline / PWA pesada
  • Juegos en el browser
  • Solo sitio estático sin JS
Pregunta clave: ¿necesito SEO, datos del servidor, o construir un producto completo? Si sí — Next.js es la opción más madura del ecosistema.
Para proyectos de clase o prototipos rápidos, Vite + React puede ser más fácil de arrancar. Pero para algo que otros van a usar — Next.js. El portafolio personal es el caso de uso perfecto para empezar.

Ecosistema — los esenciales

🎨 Estilos

  • Tailwind CSS — el estándar hoy. Clases en el JSX, sin archivos CSS.
  • CSS Modules — styles aislados por componente
  • shadcn/ui — componentes listos sobre Tailwind

🗄️ Base de datos

  • Supabase — PostgreSQL + auth + storage. Sin configurar servidor. El más fácil para arrancar.
  • Prisma ORM — cualquier DB con tipos
  • Neon / PlanetScale — DB serverless

🚀 Deploy

  • Vercel — un push a GitHub y está vivo. Zero config, plan gratis.
  • Netlify — alternativa similar
  • Railway / Render — más control
  • Docker — producción propia
Stack mínimo para lanzar algo real: Next.js + Tailwind + Supabase + Vercel. Para auth: NextAuth.js. Para estado global: Zustand.
No traten de aprender todo a la vez. Con el stack del highlight box pueden construir y lanzar un producto real. Lo demás lo aprenden cuando un problema concreto lo requiere.

Resumen

Qué ofrece Next.js

  • Routing basado en archivos
  • SSR, SSG, ISR, CSR por ruta
  • Server y Client Components
  • Server Actions — mutations
  • Route Handlers — API REST
  • Image, Font, Link optimizados
  • Zero-config con Turbopack

El modelo mental clave

  • Archivo = Ruta automática
  • Cada ruta elige su estrategia de rendering
  • Todo es Server Component por defecto
  • "use client" solo donde hay interactividad
  • Los datos fluyen del servidor al cliente
Next.js no es solo SSR — es elegir la herramienta correcta para cada parte de tu app.

¿Cuándo Vite + React?

  • Proyectos simples sin SEO
  • Dashboard interno para usuarios logueados
  • Ya tenés una API externa (Django, Laravel…)
  • Prototipo rápido de una idea
Si solo recuerdan tres cosas: 1) archivo = ruta, 2) cada ruta elige cómo renderizar, 3) server component por defecto. Con eso ya tienen el 80% del framework.

¿Qué sigue?

  1. nextjs.org/learn — Tutorial oficial, gratis, 4-6 horas. Construís una app de facturas desde cero cubriendo todo lo de hoy con más profundidad.
  2. Tu portafolio en Next.js — SSG para las páginas, Server Action para el formulario de contacto, deploy en Vercel. El mejor primer proyecto real.
  3. Next.js + Supabase — PostgreSQL + auth sin configurar servidor. Stack completo en una tarde.
  4. Usa un LLM como tutor estructurado — que te evalúe concepto por concepto, señale los vacíos y repita hasta que puedas explicarlo por tu cuenta.
La señal de que lo entendiste: podés crear un proyecto desde cero, elegir la estrategia de rendering de cada ruta y explicarle la diferencia entre Server y Client Components a alguien más.
El portafolio es especialmente valioso: lo ven los empleadores, te fuerza a tomar decisiones reales, y te genera algo concreto para mostrar.
